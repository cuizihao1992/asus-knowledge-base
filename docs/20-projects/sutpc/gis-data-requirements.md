## 总体判断

SUTPC 全生命周期监管平台的 GIS 数据需求不是简单底图需求，而是一个“空间底板 + 业务对象 + BIM/三维模型 + 监测视频 + 大屏交互”的综合数据清单。

核心要求可以概括为：

- 可看：图层能在地图或三维场景中展示。
- 可查：对象可点选、检索、查看属性。
- 可分析：支持空间统计、范围查询、路径查询、规划分析、建设管理、养护管理。
- 可联动：GIS 对象、BIM 模型、项目、合同段、设施、监测点、视频和预警能互相关联。

## 数据建模建议

如果后续把这些需求转成知识库结构，建议每个 GIS 图层按以下字段记录：

| 字段 | 说明 |
| --- | --- |
| 图层名称 | 如行政区划、路网、桥梁、隧道、监控视频 |
| 图层类别 | 基础数据、道路设施、交通设施、POI、规划数据、BIM/三维、监测预警 |
| 几何类型 | 点、线、面、三维模型、影像、点云 |
| 关键属性 | 业务字段或元数据字段 |
| 调用方式 | URL、HTTP 接口、OGC 服务、GeoJSON、3D Tiles URL |
| 可看 | 是否需要展示 |
| 可查 | 是否需要查询属性 |
| 可分析 | 是否需要空间分析 |
| 应用场景 | 大屏展示、规划分析、建设管理、养护管理、监测预警 |
| 来源系统 | 规资局、智慧建设平台、智慧管养平台、大数据平台、T-GIS、BIM 支撑平台 |
| 本地样例 | 对应 `E:\Data` 路径 |
| 风险 | 是否大文件、是否需要权限、是否需确认坐标系 |

## 当前结论

SUTPC GIS 数据需求的重点不是收集更多文件，而是把每类空间对象变成可管理的数据资产：

- 明确图层类别。
- 明确字段和几何类型。
- 明确调用方式。
- 明确可看、可查、可分析要求。
- 明确业务场景。
- 明确和 BIM/三维资产的关联。

后续最值得继续做的是 `sutpc-bim-integration.md`，把 BIM 支撑平台、T-GIS 支撑平台、大数据平台和业务系统之间的数据流单独沉淀出来。
