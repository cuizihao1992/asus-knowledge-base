# SUTPC 设施全生命周期监管需求摘要

生成日期：2026-06-20

资料来源：`E:\SUTPC\全生命周期` 下的需求规格说明书、需求调整清单、GIS/BIM 功能梳理、GIS 数据需求、大屏地图交互、BIM 数据集成方案，以及根目录中的 BIM 支撑平台汇报和智慧投资项目汇报材料。本文只提炼需求和项目经验，不移动、删除、解压或上传原始资料。

## 一句话概括

设施全生命周期监管子系统的核心，是以 BIM+GIS 为底座，把交通设施从规划、前期、建设到养护的业务流程、项目资料、空间位置、模型资产、监测数据和监管动作串起来，形成“规-建-养”跨阶段的数据传递和监管闭环。

## 项目定位

需求规格说明书中的背景是：以 BIM+GIS 为基础，打造设施多维感知、处置高效、性能评估精准、决策支持全面的设施全生命周期安全监管平台。

| 层级 | 目标 | 解释 |
| --- | --- | --- |
| 业务层 | 规划、建设、养护监管线上化 | 把常用监管流程在线办理、信息归集、跨阶段传递 |
| 数据层 | 设施监管数据数字化移交 | 规划数据、建设数据、养护数据和设施基础信息持续积累 |
| 决策层 | 支撑科学监管和决策 | 通过数据看板、GIS/BIM 可视化、监测预警和统计分析提升监管能力 |

## 业务主线

```text
规划成果 / 前期资料
  -> 建设计划 / 设计审查 / 工程监管 / 交竣工
  -> 设施库 / 合同库 / 数字化移交
  -> 管养、监测、预警、应急、考核
  -> 数据看板和 GIS/BIM 可视化
```

| 数据类 | 代表内容 | 用途 |
| --- | --- | --- |
| 项目基础数据 | 项目编号、项目名称、责任单位、阶段、总投资、计划年份、合同信息 | 项目库、前期进展、调度和看板 |
| 建设过程数据 | 合同段、开工、施工图、工程变更、计量、支付、进度计划、实际进度 | 建设监管、投资管理、工程监管 |
| 管养和设施数据 | 设施库、道路设施、桥梁、隧道、边坡、场站、养护项目 | 设施全生命周期对象管理 |

数据需求中大量字段的保存周期为 10 年，说明平台被设计为长期监管和历史追溯系统，而不是一次性展示系统。

## BIM / GIS 集成设计

| 对接对象 | 提供能力 |
| --- | --- |
| 大数据平台 | 规划、前期、建设、养护等数据写入和读取 |
| 交通业务支撑平台 | 算法请求、报表工具 |
| BIM 支撑平台 | BIM 模型导入解析、可视化、数据查询 |
| T-GIS 支撑平台 | GIS 数据导入解析、可视化、数据查询 |

BIM 侧核心流程：

```text
多源异构 BIM / GIS 数据
  -> 上传或服务接入
  -> 解析为平台标准格式
  -> 数据发布
  -> 场景编辑和编排
  -> 发布审批
  -> 业务系统通过 URL / geokey / 接口加载
```

集成方案说明，BIM 模型文件展示部分更适合放在文件存储服务器，构件属性、空间属性、模型关联业务数据放在数据库中；大文件如 3D Tiles、S3M、B3DM、glTF、OBJ、FBX、TMS 等不适合直接存关系型数据库。

## 可复用项目经验

1. 业务对象要先统一：项目库、合同库、设施库、法人库是跨阶段监管的对象基础。
2. GIS/BIM 是业务入口，不只是展示层：真正有价值的是“空间对象 + 业务属性 + 监管动作”的绑定。
3. 大文件和属性数据要分层存储：模型、切片、缓存放文件系统或对象存储；元数据、构件属性、场景编排放数据库。
4. 需求要经过处室收敛：需求调整清单显示很多功能被拆分、删除或改为对接外部平台。
5. 全生命周期需要跨阶段数据传递：规划成果、前期资料、建设过程、交竣工、转固、管养、监测预警都要明确对象、字段、来源系统、更新频率和责任单位。
