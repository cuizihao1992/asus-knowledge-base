生成日期：2026-06-20

## 一句话概括

SUTPC 全生命周期监管系统里的 BIM 不是单独的模型查看器，而是通过 BIM 支撑平台把设计、施工、运维阶段的模型文件解析成可发布、可检索、可叠加、可被 T-GIS 和业务系统调用的数据服务，再和项目、设施、进度、视频、监测、养护等业务数据建立关联。

## 平台关系

| 平台 | 角色 | 对全生命周期系统的能力 |
| --- | --- | --- |
| 大数据平台 | 数据归集和治理中心 | 提供规划、前期、建设、养护等数据写入和读取服务；接收 BIM 平台推送的结构化业务数据 |
| 交通业务支撑平台 | 业务能力组件 | 提供算法请求、报表工具等能力 |
| BIM 支撑平台 | BIM 数据接入、解析、转换、发布和场景编排中心 | 提供 BIM 模型导入解析、可视化、数据查询、轻量化、场景编辑、服务发布 |
| T-GIS 支撑平台 | GIS 空间底板和交通时空信息服务平台 | 提供 GIS 数据导入解析、可视化、数据查询，并加载 BIM 平台发布的模型服务 |
| 智慧建设/智慧管养等既有平台 | 存量业务和资产数据来源 | 提供既有 BIM、点云、倾斜摄影、智能设施和业务属性数据 |

关键判断：BIM 支撑平台和 T-GIS 支撑平台共同构成公共服务层，前端或业务模块按需调用模型、图层、属性、场景和业务数据。

## 核心数据流

```text
新增 BIM / GIS / 点云 / 倾斜摄影 / 影像等数据
  -> 上传到 BIM 支撑平台或通过服务接入
  -> BIM 平台解析、拆分、分类、转换、轻量化
  -> 生成标准三维服务和访问接口
  -> 场景编辑、配准、编排
  -> 发起场景发布审批
  -> 审批通过后发布服务
  -> T-GIS 通过 URL + geokey 加载协同展示
  -> 业务系统通过接口查询模型、构件、空间和业务属性
```

既有平台数据流：

```text
智慧建设平台 / 智慧管养平台 / 既有业务平台
  -> 接口接入或文件同步
  -> BIM 平台整理转存
  -> 发布数据访问接口
  -> T-GIS 和业务系统调用
```

## BIM 数据上传与发布

| 步骤 | 动作 | 产物 |
| --- | --- | --- |
| 文件上传 | 通过 BIM 前端上传模型文件到 BIM 服务器 | 原始模型文件进入平台 |
| 静默处理 | BIM 服务自动对模型进行拆分、分类、解析入库 | 模型元数据、文件索引、构件属性、空间属性 |
| 数据发布 | 在 BIM 前端点击发布，触发模型数据发布流程 | 可访问的三维服务 |
| 状态查看 | 通过模型状态判断发布流程是否完成 | 发布状态 |
| 分享访问 | 已发布模型通过分享功能提供 URL | 可供 T-GIS 或前端访问的服务地址 |

需要保留的关键约束：

- 模型坐标系要完整传递。
- 材质、颜色等外观信息要完整传递。
- 非几何属性要完整传递。
- 发布后的模型要有唯一标识编码，方便与智慧建设平台等业务数据关联。

## T-GIS 对接方式

BIM 平台对已完成发布的 S3M 模型，通过 `URL + geokey` 的方式访问。`geokey` 是模型唯一标识 ID，数据访问接口支持 HTTP 协议。

T-GIS 的调用逻辑：

| 步骤 | 说明 |
| --- | --- |
| 获取模型列表 | T-GIS 调用 BIM 平台模型列表接口 |
| 读取 geokey | 模型信息接口返回每个模型的唯一标识 |
| 拼接访问地址 | T-GIS 按规则自行拼接 `URL + geokey` |
| 前端加载 | T-GIS 前端通过 URL 访问 BIM 平台数据 |
| 协同展示 | 在 GIS 三维场景中叠加 BIM 模型 |

这说明 BIM 与 GIS 的集成边界比较清晰：BIM 平台负责模型处理和服务发布，T-GIS 负责空间底板、场景融合和前端协同展示。

## 既有平台同步

方案中提到的既有平台同步方式偏工程化：

| 项目 | 做法 |
| --- | --- |
| 文件同步工具 | 在文件同步服务器安装 `rsync` |
| 网络端口 | 开通 `873` 端口 |
| 访问方式 | 配置域名访问，确保 BIM 支撑平台和既有平台互相访问 |
| 同步任务 | 将既有平台 BIM 数据同步到政务云内文件服务器 |
| 后续处理 | BIM 平台对同步数据整理转存，并发布访问接口 |

风险点：这类同步涉及网络、权限、端口、域名、政务云文件服务器和数据责任边界，后续如果落地，应该单独形成接口与运维清单。

## 场景编辑与审批

| 流程 | 说明 |
| --- | --- |
| 新建场景 | 在三维场景管理菜单创建场景 |
| 编辑场景 | 进入场景编辑页面 |
| 导入模型 | 选择需要导入场景的模型数据 |
| 编排场景 | 对场景数据进行自由编排和设置 |
| 发起发布 | 场景挂接好数据后申请发布 |
| 审批流转 | 经各级审批人审批 |
| 发布成功 | 审批通过后场景正式发布 |

这个流程说明模型发布不是简单文件上线，而是带有场景编排和审批控制。对监管平台来说，这能保证领导驾驶舱、业务系统和公共服务层看到的是经过确认的空间场景。

## 存储设计

| 数据类型 | 建议存储 | 原因 |
| --- | --- | --- |
| 模型元数据 | 关系型数据库，如 PostgreSQL | 结构化、可查询、可维护 |
| 模型文件索引 | 关系型数据库 | 连接业务对象、文件路径和发布状态 |
| 场景编排数据 | 关系型数据库 | 支持场景管理、审批、版本控制 |
| 构件属性 | 数据库 | 支持属性查询、业务关联和检索 |
| 空间属性 | 数据库 | 支持定位、范围查询和 GIS 关联 |
| 业务关联数据 | 数据库或大数据平台 | 关联项目、合同段、设施、进度、养护、监测 |
| 转换后的模型文件 | 文件存储系统 | 体量大、读写频繁，不适合直接存关系库 |
| 缓存文件 | 文件存储系统 | 服务展示和 Web 加载依赖频繁读取 |

方案明确提出，大文件如 `3dtiles`、`s3m`、`b3dm`、`gltf`、`obj`、`fbx`、`tms` 以及模型展示缓存文件通常体量巨大、读写频繁，不适合存放在关系型数据库中。

## 格式转换能力

| 输入格式 | 处理方式 | 输出方向 |
| --- | --- | --- |
| `osgb` 倾斜摄影 | 解析源文件、网格划分、分层分块、首层处理、纹理压缩 | `3dtiles` |
| `ifc`、`dgn`、`rvt`、`obj`、`osg`、`shp` 等 | 先转换为内部数据格式，再进行网格划分、分层分块、纹理压缩 | `3dtiles`、`fbx`、`obj`、`gltf` 等 |
| `tif` | 解析源文件、分层分块 | `tms` |
| GIS 协作新增格式 | 在原平台能力上升级 | `UDB`、`UDD`、`UDBX`、`S3M`、`S3MX` |

平台支持的输入格式覆盖 IFC、DGN、FBX、RVT、DWG、NWD、OBJ、OSG、3DXML、CATPart、CATProduct、SDF、SDNF、STP、STEP、SKP、TCZIP、OSGB、SHP、DXF、DAE、glTF 等。输出格式覆盖 3D Tiles、B3DM、glTF、OBJ、FBX、TMS、UDB、UDD、UDBX、S3M、S3MX 等。

## 轻量化策略

| 类型 | 策略 | 作用 |
| --- | --- | --- |
| 几何转换 | 参数化描述、减少三角面 | 从源头减少顶点数量 |
| 几何转换 | 图元合并 | 减少图元数量 |
| 几何转换 | 合并纹理 | 降低纹理数量和加载成本 |
| 几何转换 | 全局网格索引、模型合并 | 支持增量更新、按需加载、渲染优化 |
| 渲染处理 | 多重 LOD | 远粗近精，提高显示效率 |
| 渲染处理 | 遮挡剔除 | 只绘制可见图元 |
| 渲染处理 | 批量绘制 | 减少绘制调用，平衡 CPU/GPU 负载 |
| 渲染处理 | 纹理压缩 | 降低显存消耗，加快渲染 |
| 渲染处理 | 网格压缩 | 减少 Web 传输体量 |

对这个项目最有价值的经验是：BIM 模型从设计软件到 Web 三维场景，中间至少经过几何转换和渲染处理两个阶段。模型能不能在浏览器里流畅展示，主要取决于这两个阶段的处理质量。

## 交互展示能力

| 能力 | 说明 |
| --- | --- |
| 资源目录 | 以项目、资源、构件的树状关系管理前端展示和快速跳转 |
| 数据检索 | 支持属性精确搜索、属性模糊搜索、空间数据检索 |
| 空间定位 | 对道路、桥梁、边坡、隧道等对象按名称快速搜索并定位 |
| 属性查看 | 点击搜索结果后查看处理后的目标属性 |
| 进度管理 | 将 BIM 构件与计划进度、实际进度关联 |
| 进度推演 | 通过滑动条模拟生命周期或施工日期变化 |
| 状态着色 | 用不同颜色提示提前、延迟、正常、未施工、施工中、已施工等状态 |
| 数据导出 | 支持将进度或统计数据导出为 Excel 二次处理 |

这一部分可以直接映射到后续的“模型资产卡片”字段：每个模型不只记录文件名和大小，还应记录是否有构件树、是否可检索、是否有进度关联、是否有业务属性、是否有场景发布 URL。

## 可复用经验

1. BIM 平台的价值不在“打开模型”，而在把模型转换成统一、可发布、可检索、可关联的空间数据服务。
2. 大文件和属性数据必须分层：模型、切片、缓存放文件系统；元数据、索引、构件属性、场景编排放数据库。
3. `URL + geokey` 是 BIM 与 T-GIS 解耦的关键方式，T-GIS 不需要直接处理原始模型文件。
4. 场景发布要有审批和版本概念，否则大屏和业务系统可能加载到未确认的模型状态。
5. 进度、成本、合同、设施、养护、监测等业务信息应该通过唯一编码关联到 BIM 构件或模型对象。
6. 轻量化不是可选优化，而是 Web 三维场景能否实际使用的前置条件。

## 当前结论

SUTPC BIM 集成方案最值得沉淀的不是某个具体软件，而是一套数据工程模式：

- 接入多源异构模型；
- 转换成统一服务；
- 通过唯一标识和业务数据关联；
- 通过 T-GIS 进入空间场景；
- 通过审批发布给业务系统和大屏使用；
- 通过轻量化保证 Web 端可用。

这套模式可以直接指导后续整理 `E:\Data\全生命周期BIM`、3D Tiles、倾斜摄影、隧道视频和 Cesium/WebGL 学习资料。
